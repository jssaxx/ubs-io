# 单副本写入性能分析

## 测试背景

| 项目 | 值 |
| --- | --- |
| 副本数 | `bio.cm.copy_num = 1` |
| IO 模型 | `sdk perf write 4 1 1024` |
| IO 大小 | 4KB |
| IO depth | 1 |
| 总数据量 | 1GB |
| 总 IO 数 | 262144 |
| 吞吐 | 239.0168 MB/s |
| IOPS | 61188.30 |
| 平均时延 | 16.34 us |

淘汰已经生效：`wCacheDiskUsedSize` 从 0 增加到 24MB/28MB，且 `WCACHE_TRACE_EVICT2DISK` 有 5660 次。本文先不分析淘汰性能，只看前台单副本写入主链路。

### `PUT_WRITE_FLOW` 拆分

| Trace | AVG(us) | 说明 |
| --- | ---: | --- |
| `WCACHE_TRACE_TIER_WRITE_META_GETSLICE` | 0.610 | 获取 meta slice |
| `WCACHE_TRACE_TIER_WRITE_META_COPY` | 0.396 | 写入 `WFlowSliceMeta` |
| `WCACHE_TRACE_TIER_WRITE_DATA_GETSLICE` | 0.675 | 获取 data slice |
| `WCACHE_TRACE_TIER_WRITE_DATA_COPY` | 0.163 | 拷贝 4KB 数据 |
| `WCACHE_TRACE_TIER_WRITE_REF_ALLOC` | 0.345 | 创建 `WCacheSliceRef` |
| 已拆分合计 | 2.189 | 仍有约 1.888us 未拆细 |

判断：`PUT_WRITE_FLOW` 当前拆出来的直接操作约 2.19us，父级 4.08us，剩余约 1.89us 可能来自函数框架、slice meta 构造、key copy、flow 地址处理、trace 开销等。

### `PUT_INSERT_INDEX` 拆分

| Trace | AVG(us) | MAX(us) | 说明 |
| --- | ---: | ---: | --- |
| `WCACHE_TRACE_INDEX_GET_TABLE` | 0.150 | 63.360 | 获取 index table |
| `WCACHE_TRACE_INDEX_HASH` | 0.128 | 10.040 | key hash |
| `WCACHE_TRACE_INDEX_LOCK` | 0.137 | 8.240 | bucket 写锁 |
| `WCACHE_TRACE_INDEX_MAP` | 0.828 | 2242.750 | map find/emplace |

判断：index 慢点主要是 `INDEX_MAP`，且 MAX 到 2242us，优先怀疑 `unordered_map` 插入分配或 rehash 抖动。

## 当前瓶颈排序

| 优先级 | 瓶颈点 | AVG(us) | 判断 |
| ---: | --- | ---: | --- |
| 1 | `WCACHE_TRACE_PUT_WRITE_FLOW` | 4.077 | server 侧最大头，优先优化 |
| 2 | `SDK_TRACE_PUT_PREPARE` | 3.893 | client 侧准备成本明显 |
| 3 | `WCACHE_TRACE_PUT_INSERT_INDEX` | 1.621 | map 插入和偶发长尾明显 |
| 4 | `MIRROR_TRACE_PUT_LOCAL_DESERIALIZE` | 0.346 | 不是主要瓶颈 |
| 5 | `WCACHE_TRACE_PUT_NEGOTIATE_QUE` / `PUT_MEM_BACK` | 0.395 / 0.474 | 单项不大，暂不优先 |

## 优化方向

| 优先级 | 优化点 | 预期收益 | 风险 | 建议 |
| ---: | --- | --- | --- | --- |
| 1 | 减少 `WCacheTier::Write()` 中每次 4KB put 的固定成本 | 降低 server 侧主耗时 | 中 | 继续拆细未解释的 1.89us，再决定是否合并流程或减少对象构造 |
| 2 | 优化 index 插入 | 降低平均耗时和长尾 | 中 | 评估 bucket map 预留容量、减少 emplace 分配、避免 rehash |
| 3 | 优化 SDK prepare get slice | 降低 client 侧 prepare 成本 | 中 | 看 `PrepareResource` 是否能批量化、复用或减少跨模块调用 |
| 4 | 减少 prepare 阶段序列化/构造成本 | 小幅降低 SDK prepare | 低 | 减少重复字段填充和 slice 描述拷贝 |
| 5 | 评估 4KB 数据 copy 路径 | 收益有限 | 低 | 当前 data copy 只有 0.163us，不建议优先做 copyfree |

## 下一步建议

先不动淘汰，单副本写入优化建议按下面顺序推进：

1. 继续细分 `WCACHE_TRACE_PUT_WRITE_FLOW` 中未解释的约 1.89us。
2. 检查 `WCACHE_TRACE_INDEX_MAP` 的分配和 rehash 情况。
3. 再看 `SDK_TRACE_PUT_PREPARE_GET_SLICE` 是否可以减少一次 put 的固定开销。
