# 单副本写入性能分析

## 测试背景

基于 2026-04-27 11:38 的 `sdk trace show` 结果分析。

| 项目 | 值 |
| --- | --- |
| 副本数 | `bio.cm.copy_num = 1` |
| IO 模型 | `sdk perf write 4 1 1024` |
| IO 大小 | 4KB |
| IO depth | 1 |
| 总数据量 | 1GB |
| 总 IO 数 | 262144 |
| 吞吐 | 211.5881 MB/s |
| IOPS | 54166.55 |
| 平均时延 | 18.46 us |

说明：这版加了更多嵌套 trace，吞吐比上一版低，绝对耗时会受打点开销影响。本文主要看相对排序和瓶颈归因。淘汰已生效，但不作为本轮前台写入优化重点。

## 主链路耗时

| Trace | BEGIN | AVG(us) | TOTAL(us) | 说明 |
| --- | ---: | ---: | ---: | --- |
| `SDK_TRACE_PUT` | 262144 | 15.181 | 3979531.870 | 单次 4KB put 总耗时 |
| `SDK_TRACE_PUT_SEND` | 262144 | 9.545 | 2502061.130 | 发送到本地 server 并等待返回 |
| `SDK_TRACE_PUT_LOCAL_SYNC` | 262144 | 9.083 | 2380991.030 | 本地同步调用 |
| `MIRROR_TRACE_PUT_RECEIVE_LOCAL` | 262144 | 8.235 | 2158737.050 | server 收到本地 put |
| `MIRROR_TRACE_PUT` | 262144 | 7.689 | 2015695.370 | mirror put 主体 |
| `WCACHE_TRACE_PUT` | 262144 | 7.359 | 1929097.080 | wcache put 主体 |
| `SDK_TRACE_PUT_PREPARE` | 262144 | 4.873 | 1277374.350 | SDK 侧准备请求 |

结论：单副本没有远端副本同步成本，主耗时集中在本地 `PUT_SEND -> LOCAL_SYNC -> MIRROR_PUT -> WCACHE_PUT`。server 侧最大头是 `WCACHE_TRACE_PUT`，client 侧最大头是 `SDK_TRACE_PUT_PREPARE`。

## WCache 瓶颈

| Trace | AVG(us) | 占 `WCACHE_TRACE_PUT` 比例 | 说明 |
| --- | ---: | ---: | --- |
| `WCACHE_TRACE_PUT_WRITE_FLOW` | 5.162 | 70.1% | 写 meta/data flow，当前 WCache 内第一瓶颈 |
| `WCACHE_TRACE_PUT_INSERT_INDEX` | 1.649 | 22.4% | 写 cache index，map 插入成本明显 |
| `WCACHE_TRACE_PUT_MEM_BACK` | 0.476 | 6.5% | 启动 mem back evict task，不优先 |
| `WCACHE_TRACE_PUT_NEGOTIATE_QUE` | 0.395 | 5.4% | 单副本下加入 evict queue，不优先 |
| `WCACHE_TRACE_PUT_GET_WCACHE` | 0.165 | 2.2% | 查找 wcache 对象，不是瓶颈 |

判断：`WCACHE_TRACE_PUT_WRITE_FLOW` 是 server 侧首要优化目标，其次是 `WCACHE_TRACE_PUT_INSERT_INDEX`。

## `PUT_WRITE_FLOW` 拆分

| Trace | BEGIN | AVG(us) | TOTAL(us) | 说明 |
| --- | ---: | ---: | ---: | --- |
| `WCACHE_TRACE_TIER_WRITE_META_GETSLICE` | 262144 | 0.918 | 240762.160 | 获取 meta slice |
| `WCACHE_TRACE_TIER_WRITE_DATA_GETSLICE` | 262144 | 0.964 | 252737.910 | 获取 data slice |
| `WCACHE_TRACE_TIER_WRITE_META_COPY` | 262144 | 0.727 | 190533.460 | 写 `WFlowSliceMeta` 到 meta slice |
| `WCACHE_TRACE_TIER_WRITE_DATA_COPY` | 262144 | 0.165 | 43132.370 | 拷贝 4KB 用户数据 |
| `WCACHE_TRACE_TIER_WRITE_REF_ALLOC` | 262144 | 0.339 | 88806.250 | 创建 `WCacheSliceRef` |
| `WCACHE_TRACE_TIER_WRITE_PREPARE_META` | 262144 | 0.155 | 40583.820 | key copy + meta 字段填充 |

结论：

| 结论 | 依据 |
| --- | --- |
| `GetSlice` 是 `PUT_WRITE_FLOW` 内部第一类成本 | meta/data 两次 get slice 合计约 1.882us |
| `META_COPY` 明显偏高 | `META_COPY 0.727us`，比 `DATA_COPY 0.165us` 高很多 |
| 4KB data copy 不是瓶颈 | `DATA_COPY 0.165us` |
| key copy/meta 字段填充不是瓶颈 | `PREPARE_META 0.155us` |

## `GetSlice` 拆分

| Trace | BEGIN | AVG(us) | TOTAL(us) | 说明 |
| --- | ---: | ---: | ---: | --- |
| `WCACHE_TRACE_TIER_GETSLICE_GETADDR` | 818735 | 0.507 | 415304.050 | `flow->GetAddrByOffset()` |
| `WCACHE_TRACE_TIER_GETSLICE_MAKE_REF` | 818735 | 0.248 | 203286.390 | `MakeRef<WCacheSlice>` |
| `FLOW_TRACE_GETADDR` | 820331 | 0.164 | 134550.920 | flow 地址构建内部耗时 |

判断：`GetSlice` 的成本不只来自 `FLOW_TRACE_GETADDR`，还包括临时 `std::vector<FlowAddr>` 构造、`GetAddrByOffset` 外围逻辑、`MakeRef<WCacheSlice>` 和引用计数对象构造。4KB 单 chunk 场景可以考虑做快路径，减少 vector 和对象构造成本。

| `WCACHE_TRACE_PUT_INSERT_INDEX` | 262144 | 1.649 | 277.190 | 432292.860 | index 插入总耗时 |
| `WCACHE_TRACE_INDEX_GET_TABLE` | 264836 | 0.151 | 56.500 | 39968.310 | 获取 index table |
| `WCACHE_TRACE_INDEX_HASH` | 264836 | 0.137 | 7.870 | 36399.730 | key hash |

| `WCACHE_TRACE_INDEX_MAP` | 264836 | 0.855 | 55.750 | 226312.550 | map find/emplace |

判断：index lock 不是主要问题，主要成本在 `INDEX_MAP`。优先怀疑 `unordered_map` 插入分配、bucket 扩容、key 比较和 emplace 成本。

| `WCACHE_TRACE_PUT_NEGOTIATE_QUE` | 0.395 | 单副本 evict queue 入队不是主瓶颈 |
| `WCACHE_TRACE_PUT_MEM_BACK` | 0.476 | 启动 evict task 不是主瓶颈 |

## 当前瓶颈排序

| 优先级 | 瓶颈点 | AVG(us) | 原因 |
| ---: | --- | ---: | --- |
| 1 | `WCACHE_TRACE_PUT_WRITE_FLOW` | 5.162 | server 侧最大头，占 WCache put 约 70% |
| 2 | `SDK_TRACE_PUT_PREPARE` | 4.873 | client 侧准备成本明显，影响端到端时延 |
| 3 | `WCACHE_TRACE_PUT_INSERT_INDEX` | 1.649 | index map 插入成本稳定存在 |
| 4 | `WCACHE_TRACE_TIER_WRITE_META_GETSLICE` + `DATA_GETSLICE` | 1.882 | 每次 put 要获取 meta/data 两个 slice |
| 5 | `WCACHE_TRACE_TIER_WRITE_META_COPY` | 0.727 | meta copy 成本明显高于 data copy |

## 优化建议

| 优先级 | 优化点 | 目标 | 建议做法 | 风险 |
| ---: | --- | --- | --- | --- |
| 1 | 优化 `WCacheTier::GetSlice()` 4KB 快路径 | 降低 `META_GETSLICE` 和 `DATA_GETSLICE` | 避免每次构造临时 `std::vector<FlowAddr>`；4KB 单 chunk 场景可使用固定小数组或直接构造单地址 slice | 中 |
| 2 | 优化 `WCacheSlice` / `WCacheSliceRef` 对象构造 | 降低 `GETSLICE_MAKE_REF` 和 `REF_ALLOC` | 评估对象池、减少引用计数对象分配、合并 slice/ref 构造路径 | 中 |
| 3 | 优化 meta 写入路径 | 降低 `META_COPY 0.727us` | 检查 `mSliceOperator.Copy` 对 memory slice 的路径；若只是内存写，考虑直接写入或减少安全拷贝/地址解析成本 | 中 |
| 4 | 优化 `WCacheIndex::Insert()` | 降低 `PUT_INSERT_INDEX` | 预留 map 容量，减少 rehash；或利用 `indexInFlow` 递增特点改成更直接的索引结构 | 中 |
| 5 | 优化 SDK prepare | 降低端到端 `SDK_TRACE_PUT` | 优先看 `PrepareResource` 和 `DataCopy`；评估批量申请、复用请求对象、减少 slice 描述序列化 | 中 |
| 6 | 保留 data copy 现状 | 避免无效优化 | 当前 `DATA_COPY` 只有 0.165us，copyfree 不是优先方向 | 低 |

## 下一步建议

| 步骤 | 内容 | 预期产出 |
| ---: | --- | --- |
| 1 | 看 `GetSlice()` 4KB 单 chunk 快路径可行性 | 明确是否能减少 vector 和对象构造 |
| 2 | 单独分析 `META_COPY` 的 `CacheSliceOperator::Copy` 路径 | 判断 0.727us 来自安全拷贝、地址解析还是 trace 开销 |
| 3 | 给 index map 做 reserve 或替代结构实验 | 验证 `INDEX_MAP` 是否能下降 |
